# Why RepliBuild

## The problem

Calling C/C++ from Julia means crossing an ABI boundary that the source language cannot describe. Headers declare types in human-readable form, but the actual in-memory layout — field offsets, padding, alignment, vtable slots, bitfield positions, enum sizing, calling-convention return-by-value rules — is decided by the compiler at compile time, against a target triple, a set of attributes, and a packing pragma stack. A header is a recipe; the binary is the cake. Writing FFI by reading the recipe means guessing the cake.

For a single scalar function with two `int` arguments this gap is invisible. For real libraries — hundreds of functions, packed structs in network protocols, bitfields in hardware registers, unions in tagged variants, virtual methods through inheritance chains, alignment that differs across platforms and compiler versions — every guess is a chance for silent corruption. A struct that reads `{ char tag; int value; }` looks identical in source whether it is six bytes packed or twelve bytes aligned. A `Cint` written where the binary expects a `Clong` will silently scramble subsequent fields. There are no compile-time diagnostics for this; failures surface as garbage values or segfaults far from the actual mismatch.

RepliBuild takes a different approach: compile the source, then extract what the compiler knows. The compiler has already done the work of resolving every `#ifdef`, applying every `__attribute__((packed))`, computing every alignment for the target platform, and laying out every vtable. That information is preserved in DWARF debug metadata. RepliBuild reads it and emits Julia bindings whose struct layouts, field offsets, function signatures, and dispatch tables are byte-for-byte identical to what the compiled binary expects.

## DWARF as a stable source of truth

DWARF is the debug information format that LLVM, GCC, and other compilers emit alongside compiled binaries when `-g` is set. It is not a separate description of the source — it is the compiler's own record of what it produced. Every field that ends up in the final `.so` has a corresponding DWARF entry recording its name, type, and byte offset within its parent struct. Every function has its signature and parameter types. Every virtual method has its vtable slot. The metadata is generated by the same code path that decides the layout, so it cannot disagree with the binary it accompanies.

This gives RepliBuild three properties that header parsing alone cannot provide.

### Offsets are observed, not inferred

For each `DW_TAG_member` entry the compiler emits a `DW_AT_data_member_location` attribute holding the byte offset of that field within its enclosing struct. RepliBuild reads this offset directly. No re-derivation of layout rules, no platform-specific padding heuristics, no `__attribute__` interpretation — the value in the metadata is the value the compiled code uses to access that field.

A packed struct like:

```cpp
#pragma pack(push, 1)
typedef struct PackedTriplet {
    char tag;       // DW_AT_data_member_location = 0
    int value;      // DW_AT_data_member_location = 1
    char flag;      // DW_AT_data_member_location = 5
} PackedTriplet;    // DW_AT_byte_size = 6
#pragma pack(pop)
```

emits `DW_AT_data_member_location` values of 0, 1, and 5 — not 0, 4, and 8 that a natural-alignment guess would produce. RepliBuild emits a Julia struct with matching offsets, and because Julia would naturally align the `Int32` to byte 4, the wrapper generator detects the size discrepancy (DWARF reports 6 bytes; Julia's aligned size is 12 bytes) and routes calls through an MLIR thunk that marshals between the two layouts. The packed/aligned distinction is automatic — never declared, never guessed.

### Stability across platforms and compiler flags

The DWARF metadata is produced by the same toolchain invocation that produces the `.so`. When the target changes — different platform, different optimization level, different `__attribute__` set, different `#pragma pack` boundary, different C++ ABI version — both the binary and its DWARF change together. The wrapper that RepliBuild generates against a given binary is correct for that binary specifically. There is no "the wrapper assumed natural alignment, but this build used `-fpack-struct`" failure mode, because the wrapper never assumed anything; it read the offsets from the binary's own metadata.

This also means the wrapper survives compiler upgrades. If a Clang revision changes how it lays out a particular tail-padding edge case, the new DWARF reflects the new layout, and a regenerated wrapper picks it up automatically. RepliBuild does not encode layout rules; it encodes a procedure for reading them.

### Coverage of constructs not present in source

DWARF carries information that source code does not directly express:

- **Vtable slots.** `DW_TAG_subprogram` entries inside a class definition with the `DW_AT_virtuality` attribute list every virtual method along with its slot index in the vtable. RepliBuild reads this to generate `jlcs.vcall` operations that index into vtables by exact slot, matching the dispatch behavior of a C++ caller.
- **Bitfield positions.** `DW_AT_bit_offset` and `DW_AT_bit_size` record the bit-level placement of bitfields inside their storage units. RepliBuild emits getter/setter accessors that shift and mask the underlying integer at the correct positions.
- **Inheritance hierarchies.** `DW_TAG_inheritance` records base classes with their member-offset within the derived class, allowing RepliBuild to flatten base-class fields into the Julia struct definition or emit upcast accessors.
- **Underlying enum types.** `DW_TAG_enumeration_type` records both the named values and the integral type the compiler chose for them (which may differ from a default `int` if the compiler optimized to a smaller type).
- **Sret conventions.** Return-by-value structs that exceed the platform's small-return threshold are recorded with explicit sret parameter positions, which RepliBuild reproduces in its calling convention.

None of this can be derived from headers alone. All of it is present in DWARF as direct, unambiguous values.

## Combining DWARF with other sources

DWARF is the backbone but not the whole picture. RepliBuild combines four information sources to produce bindings that are correct by construction:

**DWARF debug metadata** (the structural backbone):

- Struct member offsets, sizes, and byte positions — exactly as the compiler laid them out
- Function return types and parameter types
- Class inheritance and virtual method information with explicit vtable slots
- Bitfield bit-offsets and widths
- Union member layouts and total size
- Global variable types and storage
- Typedef chains and template instantiations

**Symbol table** (`nm`) — the linking identity:

- Mangled C++ names — the authoritative linking identity that DWARF's demangled names map to
- Function addresses in the compiled binary
- Vtable symbol addresses for virtual dispatch verification
- Distinction between defined symbols, undefined imports, and weak symbols

**Clang.jl AST** — fills the gaps DWARF cannot:

- Enums that the compiler optimized away (unused in compiled code, so absent from DWARF)
- Function pointer typedefs that never appear as concrete function parameters
- Preprocessor macros (via typed shims declared in `[wrap.macros]` which become real functions and thus appear in DWARF)

**Cross-verification** — catches the cases where one source is incomplete:

- DWARF-reported struct size vs Julia's computed aligned size — detects packed layouts and routes them through MLIR thunks
- Symbol presence in `nm` vs DWARF subprogram entries — flags any function declared in metadata but missing from the binary, and vice versa
- DWARF type chains against AST type strings — catches typedef resolution disagreements

When a struct is `#pragma pack(1)`, the header looks identical to its aligned counterpart — same field names, same field types. But DWARF records `member b at DW_AT_data_member_location = 1` instead of `= 4`, and `DW_AT_byte_size = 6` instead of `= 12`. RepliBuild sees this, flags the struct as packed, and generates a safe MLIR thunk instead of emitting a broken `ccall`.

## Three-tier dispatch

Every function gets automatically routed to the right calling mechanism based on what its DWARF signature requires:

| Tier | Mechanism | When | Cost |
|------|-----------|------|------|
| **1** | `Base.llvmcall` with LTO bitcode | POD args, scalar/pointer return, bitcode available | Zero — C/C++ inlines into Julia's JIT, cross-language optimization |
| **2** | MLIR thunks (`libJLCS.so`) | Packed structs, unions, large struct return, virtual dispatch | ~40 ns per call; first call ~500 µs for JIT (or zero with AOT) |
| **3** | `ccall` | Fallback when bitcode unavailable | ~40 ns per call |

Tier selection is fully automatic. RepliBuild analyses each function's DWARF metadata, checks if `ccall` is safe (alignment, size, type complexity), and emits the appropriate wrapper. If the struct layout doesn't match what Julia would expect, it goes to MLIR. If LTO bitcode is available and the types are simple, it goes straight through `llvmcall` with zero FFI overhead. The user never annotates anything; the decision falls out of the metadata.

## When to use RepliBuild

RepliBuild is built for developers who want hands-on, source-driven wrapping of C/C++ libraries — typically when you have a build tree you can run locally, a `CMakeLists.txt` or set of source files you control, or a debug-built binary with DWARF present.

It fits well when:

- You have C/C++ **source code** or a debug-compiled binary with DWARF metadata
- You want ABI-correct bindings without writing struct definitions by hand
- You need C++ virtual method dispatch from Julia, including dynamic dispatch through base-class pointers
- You want LTO — C/C++ code inlined into Julia loops, optimization across the FFI boundary
- Your library uses packed structs, bitfields, unions, varargs, or complex calling conventions
- Your library evolves and you want regenerated wrappers to track ABI changes without manual intervention

**Pre-compiled C binaries can be ingested (experimental).** If you have a **C** `.so`/`.dylib` compiled with debug info (`-g`), `ingest()` can wrap it without recompiling — DWARF extraction and wrapper generation still run, dispatching through Tier 3 (`ccall`) only, since LTO and Tier 2 thunks both require source-stage compilation. This is a best-effort fallback, not the flagship path: the binary was produced by a compiler and debug-info configuration RepliBuild doesn't control, so extraction quality varies. **C++ API surfaces are not supported in ingest mode** — classes, methods, templates, and virtual dispatch need the MLIR dialect thunks only the source build generates. For C++ libraries, build from source with RepliBuild (or ingest their C API variant).

## How the pieces fit together

RepliBuild compiles your C/C++ source with Clang (`-g` for debug info), then runs a five-stage pipeline that turns the resulting binary + metadata into a Julia module:

```
source → Compile → Link → DWARF extract → Tier dispatch → Wrap
```

1. **Compile** — Each source file is compiled to LLVM IR (`.ll`) via `clang` or `clang++`. Per-file `mtime` cache, parallel dispatch, IR sanitization to match Julia's internal LLVM version. Template instantiations and macro shims are auto-generated so they appear in DWARF.
2. **Link** — Sanitized per-file IR is merged via `llvm-link`, optimized by `llvm-opt`, and linked into a shared library. Optional artifacts include the LTO bitcode (`_lto.bc` for Tier 1 `llvmcall`) and pre-compiled MLIR thunks (`_thunks.so` for Tier 2 AOT).
3. **DWARF extract** — `llvm-dwarfdump` output is parsed into structured Julia types: `ClassInfo` (with virtual methods and inheritance), `VtableInfo`, `MemberInfo` (with byte offsets), and others. `nm` provides the mangled symbol table for linking identity.
4. **Tier dispatch** — `is_ccall_safe()` and `is_c_lto_safe()` inspect each function's DWARF signature to decide which tier handles it. Packed-struct detection, STL container detection, union detection, and return-by-value size analysis all feed this decision.
5. **Wrap** — The wrapper generator emits a complete Julia module: struct definitions with correct field order and padding, `@enum` declarations, function wrappers dispatched to the appropriate tier, bitfield accessors, `mutable struct` idiomatic wrappers with GC finalizers for factory/destructor pairs, and `cglobal` accessors for globals.

Tier 2 functions don't compile their thunks until the first call (or build time, with `aot_thunks = true`). The JIT manager caches compiled symbol pointers in a lock-free dictionary so subsequent calls are a single unsynchronized `Dict` read.

## The JLCS dialect

Some ABI situations cannot be expressed in `ccall` at all — packed struct field access at the IR level, virtual method dispatch through vtables with explicit slot indices, strided multi-dimensional array views, RAII scopes that destruct C++ objects in reverse construction order, C++ exception catching with landing pads. Rather than emit fragile inline LLVM IR for these, RepliBuild lowers them through a small, purpose-built MLIR dialect.

The JLCS (Julia C-Struct) dialect models C ABI contracts as first-class IR operations: `jlcs.c_struct<>` types carry field types, byte offsets, and packing flags; `jlcs.vcall` encodes vtable offset and slot index; `jlcs.ffe_call` marshals arguments through the ciface convention; `jlcs.scope` regions auto-emit destructors in reverse order. These operations are verified by MLIR's standard machinery, lowered through the LLVM dialect, and JIT-compiled to native code. The dialect is small (under a dozen operations) and exists only to express what regular LLVM IR cannot.

For full operation semantics see [MLIR / JLCS Dialect](mlir.md).

## Templates, varargs, and macros

DWARF only records constructs the compiler actually instantiated. Template specializations that are never used, variadic functions that the C standard does not allow to inspect, and preprocessor macros (which exist only at the preprocessor stage and never reach the AST) are all invisible to a pure-DWARF approach. RepliBuild handles these through declarative configuration that forces them into the compiled artifact:

```toml
[types]
templates = ["std::vector<int>"]       # auto-generates a stub .cpp that instantiates this template,
template_headers = ["<vector>"]        # forcing Clang to emit DWARF for the instantiation

[wrap.varargs]
printf = [["Cstring", "Cint"], ["Cstring", "Cdouble"]]   # typed overloads for variadic calls

[wrap.macros]
MAX_SIZE = { type = "Cint" }           # generates a typed C shim function so the macro
                                       # becomes a real symbol with DWARF metadata
```

Each declaration causes RepliBuild to emit code that materializes the construct as a real symbol in the binary, which then appears in DWARF, which then becomes part of the Julia wrapper. The pattern is consistent: anything not in DWARF is made to be in DWARF.

## Summary

RepliBuild does not parse C/C++ source. It compiles it and reads what the compiler produced. The compiler has already resolved every `#ifdef`, every alignment rule, every platform-specific calling convention, and recorded it in DWARF as concrete byte offsets, type sizes, and vtable slots. RepliBuild reads those values and emits Julia code that matches them exactly. When DWARF is incomplete, configuration directs additional symbols into the compiled artifact so they appear in subsequent DWARF passes. When the ABI is too complex for `ccall`, a small MLIR dialect bridges the gap through verified IR lowering.

The result is a binding generator whose correctness depends on the same toolchain that produced the binary, rather than on the wrapper author's knowledge of platform alignment rules. You need source code or a debug-compiled binary, and an LLVM 21+ toolchain. With source you get the full pipeline (LTO, AOT thunks, incremental builds). With a debug binary you get DWARF-correct wrappers and MLIR thunks. Either way, the entire class of layout-mismatch FFI bugs disappears.
